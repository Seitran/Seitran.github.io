网络流控之普通流控与PFC流控

发表于 2015-10-24 分类于 技术分享

今天跟大家讲两个故事，请忽略故事背景和故事的合理性。

1.有两个小伙伴，小胖和小强。小胖喜欢吃东西，胃口也大，每次都会给小强很多吃的和喝的。这天，小胖不断地给小强饼干和可乐。小强一直吃啊吃啊，最后发现自己快撑了，吃不动了。就告诉小胖说，“我吃不了啦，你不要再给我东西了！”于是小胖就停下来了，不再去给小强吃的了，小强就慢慢地把小胖送过来的吃的一个一个给消化掉。

这个场景，很像网络中的普通MAC流控。当链路中发生拥塞时，接收的一方就会告诉另外一端，要求其暂停发送，休息一下。这个时候，发送方整个网络端口的流量都会被暂停掉。

2.同样的场景，小胖不断仍然不断地给小强传递饼干和可乐。小强这个时候发现自己已经吃不下饼干了，但是可乐呢，依然还能够喝得下。小强冲着小胖吼了一吼，“不要再给我饼干了！”于是小胖就不再给小强饼干了，但是可乐依然源源不断的递过来。

这个场景，很像网络中的PFC流控。当链路发生拥塞时，PFC流控可以基于优先级，来对不同的流量进行控制，暂停某一种类型的流量，而不会影响到其他类型的流量。

关于PFC，可以展开说一些。实际上，PFC是DCB（数据中心桥）协议族的一个子协议，主要用在数据中心的场景下。PFC是基于优先级的流量控制，最大能够支持的优先级为8，正好对应着vlan中的3bit的优先级字段，也就是说PFC是应用于vlan场景的。对于支持PFC的硬件而言，

一般会有8个traffic class（TC），每一个TC一般对应着一个优先级（也可以多个优先级存在于一个tc中）。另外，优先级这个概念与vlan没有对应关系。一个vlan可以有多个优先级，同样的，同一个优先级也可以对应于多个vlan。

ethtool工具dump寄存器显示的问题

发表于 2015-07-29 更新于 2016-01-28 分类于 技术分享

工作中，发现dump寄存器的时候,一部分比较出名的网卡，比如intel 82599，会列出比较详细的信息，比如寄存器的名字、偏移以及对应的值。而有另外一些不太“出名”的网卡，仅仅只能打印寄存器的十六进制字节流。

一开始，以为是网卡驱动能够控制的，后来才发现其实并不是这个样子的。驱动对上层只提供统一的ethtool的dump寄存器的接口。而如何显示驱动提供的这些值，则是用户态的ethtool所要做的事情。
查看ethtool－4.0的代码，可以看到在ethtool.c中定义了一个数组，数组中列出了当前ethtool版本所“支持”的网卡。对于这些网卡，ethtool在dump的时候会采用其提供的函数:

比如，82599网卡久对应了ixgbe_dump_regs这个函数。

用户在使用ethtool －d ethx时，如果没有指定强制要求16进制打印，那么ethtool会在driver_list中根据驱动名称（驱动名称通过get_drvinfo获取）进行匹配，如果匹配到，则会调用对应的处理函数；否则，就会调用默认的16进制打印函数进行寄存器打印；否则，则直接进行16进制打印。

通过这一过程可以看到，这部分的设计时，已经充分考虑到数据（data）和其视图（view）的分离：驱动只负责关键数据的获取、维护、设置等。而如何向用户进行呈现，则是用户态程序的事情。这样做有几个好处：
1，这样的设计使得模块间的接口单一，模块间的交互不需要考虑各种情形，只需要简单地交换数据即可；
2，也使得各个模块的功能简单：数据的管理在底层，上层负责展示，不会使功能混杂在一个模块中；
3，这样设计也易于扩展：更改展示的代码，即可以不同的方式展示数据，而不需关心底层的硬件是怎么样子的；反之亦然，硬件的变动也不影响上层的数据展示。

jvm中使用更多的堆内存

发表于 2015-05-03 更新于 2015-05-04 分类于 技术分享

前几年给妹子的论文写了一个java版本的实验代码，现在论文修改意见回来了，需要重新做实验。当年由于实验环境比较挫，当数据集较大的时候（1M点集）就总是Out Of Memory，进行了许多尝试也没有改变这个现状。现在又需要跑这么大数据集，但有了更好的实验条件，我们是不是可以做点什么，让代码支持更大的数据集呢？

我们知道，java和c/c++最大的区别是：java是解释型语言，编译只会将其编译为字节码。运行时，由java虚拟机（JVM）对字节码进行解析，最终形成机器码被目标机器所真正执行；而C/C++是编译型语言，在编译结束时，源程序的机器码已经包含在目标文件中。

在Java中，目标程序所能够使用的最大堆内存是由Java虚拟机所管理的。默认情况下，最大的堆内存为物理内存的四分之一。在使用java运行程序时，-Xmx参数可以更改默认的最大可用的堆内存。例如-Xmx512m可以指定最大的堆内存为512M。

虽然我们的物理内存通常会比较大，而且32位机器上的虚拟内存也能够达到4G，但是JVM所能够使用的最大堆内存却远小于4GB或者实际可用的物理内存。这主要是由于虚拟内存4G中只有2G是用户态空间（通过设置内核也可使用用户态空间扩大为3G），而在用户态仅有的2G内存中，分配给堆内存的可用空间会更少。经过简单的尝试，-Xmx1400m还是可以达到的。

当我们的程序想要用到更大的堆内存时怎么办呢？32位系统下4G的虚拟内存限制是一个不可逾越的限制。64位系统正是我们所需要的。

java命令中，可以使用-d32和-d64来指定所运行的环境位宽。目前在windows平台下jdk1.5,JDK1.6不支持-d32,-d64的选项，jdk1.7支持-d32,-d64的选项。

分享一个开源项目——ksocket

发表于 2015-04-15 更新于 2016-01-28 分类于 技术分享

今天比较忙，锻炼完身体，看了《两个月亮》，就没多少精力了。今天就分享工作中使用的一个开源项目ksocket。

按照官方的介绍，ksocket是可以运行于linux 2.6内核下的一个开源的内核态BSD like的socket套接字接口。使用它，可以很方便地在内核中使用socket建立网络连接。实际上，linux 3.4内核下这个项目也是能够完美运行的，其他版本的linux内核没有测试过，但大体上应该都能够运行。

ksocket有什么用呢？首要的一点是：使用ksocket建立的内核态网络应用，能够很方便地使我们能够从用户态访问用户态。类似的工具还有netlink，但与netlink不同的是，ksocket允许我们从网络上的任意位置访问内核，而netlink只能够从本机访问。

项目中用到了一个基于ksocket的内核态rpc组件，但是该组件还不是太完善，在某些场景下会引起内核的oops。打算近期重构一个类似的基于ksocket的内核态rpc组件，锻炼一下自己的内核态代码开发能力。

网卡TSO/GSO/LRO/GRO简要介绍

发表于 2015-04-13 更新于 2016-01-28 分类于 技术分享

之前还一点儿也不了解网卡还有一些高级特性，在工作中的一次调试中，发现期望对端发送的一个报文长度居然超过了网络中的mtu（默认1500Bytes），能够达到将近3000Bytes。

一开始就排除了对端协议栈下到网卡上的数据会大于mtu（这一点在之前的工作中我使用libpcap验证过，一旦发送的报文长度大于1个mtu，发送就会失败）。然后查看虚拟网桥上抓到的包，发现网桥捕获的数据包的确大于了1个mtu。当时一直认为出现问题的地方可能是网桥在转发时聚合了数据包，后来经过同事的提醒，才认识到可能是网卡本身存在某种数据聚合的特性，使得上层协议栈所发送的数据包被聚合成了一个大包，然后被发送出去。

查阅了相关资料，这种现象实际上涉及到了网卡的GRO/GSO/LRO等特性，下面简要介绍一下。

##GSO（generic-segmentation-offload）/ TSO（TCP-segmentation-offload）
所谓的GSO，实际上是对TSO的增强。TSO将tcp协议的一些处理下放到网卡完成以减轻协议栈处理占用CPU的负载。通常以太网的MTU是1500Bytes，除去IP头（标准情况下20Bytes）、TCP头（标准情况下20Bytes），TCP的MSS (Max Segment Size)大小是1460Bytes。当应用层下发的数据超过了mss时，协议栈会对这样的payload进行分片，保证生成的报文长度不超过MTU的大小。但是对于支持TSO/GSO的网卡而言，就没这个必要了，可以把最多64K大小的payload直接往下传给协议栈，此时IP层也不会进行分片，一直会传给网卡驱动，支持TSO/GSO的网卡会自己生成TCP/IP包头和帧头，这样可以offload很多协议栈上的内存操作，checksum计算等原本靠CPU来做的工作都移给了网卡。

##GRO（generic-receive-offload）/ LRO（large-receive-offload）
LRO通过将接收到的多个TCP数据聚合成一个大的数据包，然后传递给网络协议栈处理，以减少上层协议栈处理 开销，提高系统接收TCP数据包的能力。

而GRO的基本思想跟LRO类似，克服了LRO的一些缺点，更通用。后续的驱动都使用GRO的接口，而不是LRO。

具体到工作中遇到的那个问题中，我们在关闭了网卡的GSO/TSO功能后，还是能收到大包，这里面实际上还有LRO/GRO的“功劳”在里面，将它们也关闭后，对端协议栈发送什么样的数据包，本端就收到了什么样的数据包了。

##了解网卡是否支持
在linux下，我们可以使用ethtool工具查询网卡是否支持这些特性（以网卡eth0为例）：

$ ethtool -k eth0

而我们要关闭某一个特性时，只需要输入（以关闭gso为例）：

$ ethtool -K eth0 gso off

上面的命令只能暂时关闭网卡的特性，当重启之后网卡又会启用该特性，为了避免这种情况，我们只需要将上面的命令保存到网卡的配置文件里面即可，对于我的CentOS 6.5来讲，对应于eth0的配置文件在

/etc/sysconfig/network-scripts/ifcfg-eth0
